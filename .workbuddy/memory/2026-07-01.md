## MediaServer 直播功能方案咨询（Ask 模式，未改代码）
- 用户引用 `@D:/MediaServer`（与工作区 MediaPlay 配套的短视频服务端），问"基于现有代码怎么实现直播功能，只讲方案不动代码"。
- 通读 MediaServer 全部源码摸清架构：Linux C++ 服务器，epoll + 线程池(生产者-消费者, EPOLLONESHOT) + 自定义二进制协议(4字节长度头+结构体, _DEF_PACK_BASE=10000) + MySQL(单连接+互斥锁)，现有功能注册/登录/上传(分块)/下载(分块推流式 while fread 1KB send)/上传历史。
- 关键判断：现有 DownloadRq 是「点播拉流·单连接独占线程·阻塞send·文件驱动」，直播需「推-转模型·1对多·房间化·实时驱动」，不能直接复用 DownloadRq 循环发送，但 epoll/线程池/协议格式/SendData/MySQL 全可复用。
- 给出方案：packdef.h 从 10011 起新增 9 个直播协议(START/STOP/JOIN/LEAVE/BLOCK/LIST)；CLogic 新增 LiveRoom{hostFd,viewerFds,recentBuf} + roomId/fd 双向映射；转发核心=收到主播块遍历 viewerFds 逐个 send，失败踢人。
- 冲突点：阻塞 send 多观众转发会卡死→第一版用 SO_SNDTIMEO 兜底；主播断线要在 close 回调销毁房间；H.264 关键帧(isKeyFrame)观众进入需补帧；帧>1024块需按帧重组(frameEnd标志)。
- 客户端复用：主播端复用 recorderdialog 采集编码改写文件为发包；观众端复用 playerdialog 解码+SDL2，数据源换成 LIVE_BLOCK_RQ。
- 渐进路径：协议+房间结构→信令→转发(单主播单观众)→客户端接入→优化。
- 用 Visualizer 画了两张灰度图：直播数据流架构、直播协议交互时序(9步)。
