# MediaServer / MediaPlay 项目面试答案

---

## 1. 项目用到的各种协议

项目用到了两类协议：**自定义私有 TCP 协议** 用于客户端与服务端之间的业务通信，**RTMP 协议** 用于视频的推流与点播播放。

### 自定义二进制 TCP 协议

用于客户端与服务端之间的所有业务交互，没有复用 HTTP，而是直接在 TCP 之上封装私有包格式。包头是固定的 4 字节总长度 + 4 字节包类型，后跟具体业务结构体，定义在 `packdef.h` 中。协议类型从 10000 起编号：注册（10000/10001）、登录（10002/10003）、上传元信息（10004/10005）、文件块传输（10006，每块固定 1024 字节）、下载请求/响应（10007/10008）、上传历史（10009/10010）。服务端用 epoll 驱动的事件循环接收数据，通过函数指针数组按类型 O(1) 派发到对应处理函数。

### RTMP 协议

这是项目最核心的流媒体协议，贯穿从录制、存储到播放的完整链路。

**录制侧**：客户端用 FFmpeg 将音视频编码为 H.264+AAC，封装成 FLV 格式（`savevideofilethread.cpp` 第 419 行：`avformat_alloc_output_context2(&oc, NULL, "flv", filename)`，注释明确写着"为了未来推流 rtmp（flv）"）。FLV 是 RTMP 的原生载体格式，两者天然绑定。

**服务端存储**：视频上传后，服务端在 `clogic.cpp` 第 144 行构造 RTMP 相对路径并写入数据库：

```cpp
sprintf(info->m_szRtmp, "//%s/%s", info->m_UserName, info->m_szFileName);
// 存入 t_VideoInfo 表的 rtmp 字段
```

**播放侧**：客户端下载视频列表时（`onlinedialog.cpp` 第 261 行），从服务端返回的 `m_rtmp` 字段拼出完整 RTMP 播放地址：

```cpp
info->rtmpUrl = QString("rtmp://%1/vod%2").arg(DEF_SSERVER_IP).arg(rs->m_rtmp);
// 最终形如 rtmp://192.168.43.239/vod//username/filename.flv
```

该 URL 存在 `MovieLabel` 的 `m_rtmpUrl` 字段，用户点击视频后 emit `SIG_PlayUrl(pb_play->rtmpUrl())`，播放器通过 FFmpeg 的 `avformat_open_input` 直接打开这个 RTMP 地址拉流解码。`playerdialog.cpp` 里也保留了两条调试地址：`rtmp://192.168.43.239:1935/vod//101.mp4` 和 `rtmp://192.168.43.239:1935/videotest/100`，说明本地搭了 RTMP 服务器（通常是 Nginx+nginx-rtmp-module）来做验证。

## 2. 视频预览怎么实现

视频预览分两个场景。

**录制时的摄像头预览**：录制功能要实现画中画效果，采集线程 `PicInPic_Read` 每采到一帧摄像头画面，就 emit `SIG_sendPicInPic` 信号，UI 里的 `PictureWidget` 控件接收后直接调 `slot_setImage` 刷新显示。这样摄像头画面单独呈现在一个小窗里，录制者可以实时监看。而桌面录制流保持独立，不需要把两路画面合成，避免了额外的图像合成开销。连接关系在 `recorderdialog.cpp` 第 16 行。

**视频列表的缩略图预览**：上传视频时，客户端把 GIF 缩略图文件和视频文件一起发给服务端，服务端把 GIF 的路径存入数据库 `t_VideoInfo` 表的 `picPath` 字段。客户端请求历史视频列表时，服务端把 `picPath` 随列表数据一起下发，客户端用自定义的 `MovieLabel` 控件加载这个路径的 GIF 直接显示，点击后再发起完整的视频播放请求。

## 3. 关键帧是怎么定义的

在编码侧，关键帧由 GOP（Group of Pictures）大小决定。在 `savevideofilethread.cpp` 第 293 行：

```cpp
c->gop_size = 12; /* emit one intra frame every twelve frames at most */
```

帧率设置为 25fps，所以每 12 帧出一个 I 帧（关键帧），即约每 0.48 秒一个关键帧。关键帧是可以独立解码的完整帧，不依赖前后帧；非关键帧（P 帧、B 帧）依赖参考帧做差分编码。

在解码/跳转侧，`videoplayer.cpp` 第 487 行的注释说明了关键帧在 seek 中的作用：

```
// 关键帧--比如 10 秒 --> 15 秒 跳转关键帧 只能是 10 或 15，
// 如果你要跳到 13，做法是跳到 10 然后 10-13 的包全扔掉
```

即 seek 时只能跳到关键帧位置，通过 `av_seek_frame(..., AVSEEK_FLAG_BACKWARD)` 向前找最近的关键帧，然后逐包解码丢帧直到目标时间点，这是标准的 FFmpeg seek 实现方式。

## 4. 用 FFmpeg 的时候遇到过什么问题

**问题一：seek 后花屏**。视频是按 GOP 编码的，P 帧和 B 帧依赖前面的 I 帧作为参考。如果 seek 落点不在关键帧上，解码器拿到的是 P/B 帧，但没有对应的 I 帧参考数据，输出的画面就会花屏或错乱。解决方式是用 `av_seek_frame(..., AVSEEK_FLAG_BACKWARD)` 强制跳到目标时间点之前最近的关键帧（`videoplayer.cpp` 第 500 行），再调 `avcodec_flush_buffers` 清掉解码器内部的旧状态（第 95 行），然后从关键帧开始解码，丢掉时间戳早于目标点的帧，直到追上目标位置。代码注释也说明了：比如要跳到 13 秒，实际是跳到 10 秒的关键帧，再把 10 到 13 秒的包全丢掉。

**问题二：音频格式不兼容**。FFmpeg 解码出来的音频是 `AV_SAMPLE_FMT_FLTP`（浮点平面格式），SDL 只能播放 `AUDIO_S16SYS`（16 位有符号交错格式），格式不匹配直接播放就是噪声。需要用 `swr_alloc_set_opts` 创建重采样上下文，再调 `swr_convert` 做格式转换，对应 `videoplayer.cpp` 第 752-765 行。

**问题三：中文路径打开失败**。`avformat_open_input` 在 Windows 上对非 ASCII 路径处理有问题，直接传 `QString` 转出来的 GBK 字符串会失败。解决方式是先把 `QString` 转成 `std::string`（UTF-8 编码）再传入，`videoplayer.cpp` 第 316-317 行做了这个转换。

## 5. 软解码和硬解码

软解码是用 CPU 跑解码运算，兼容性好但 CPU 占用高；硬解码是调用 GPU 或专用硬件芯片来解码，速度快、CPU 占用低，但依赖具体的硬件和驱动，比如 Windows 上的 D3D11VA、Linux 上的 VAAPI/VDPAU。编码同理，软编码用 CPU、硬编码用 GPU（NVIDIA 的 NVENC、AMD 的 AMF）。

项目用的是软解码和软编码。

软解码的流程是这样的：先用 `avformat_open_input` 打开文件，`avformat_find_stream_info` 探测流信息，然后遍历所有流找到视频流和音频流的下标。找到之后拿到对应的 `codec_id`，调 `avcodec_find_decoder` 查找解码器——这里没有指定硬件解码器名称，FFmpeg 默认就返回软件实现——再调 `avcodec_open2` 打开解码器。

解码阶段分两条线并行跑。主线程循环调 `av_read_frame` 从文件里读压缩包，根据 `stream_index` 判断是视频包还是音频包，分别放进两个 `PacketQueue` 队列。视频解码在独立的 `video_thread` 里，从队列取包后调 `avcodec_decode_video2` 解出 YUV 帧，YUV 是视频编码的原始颜色空间，但屏幕显示需要 RGB，所以还要调 `sws_scale` 做颜色空间转换，把 YUV420P 转成 RGB32，再包成 QImage 通过信号发给 UI 渲染。音频解码在 SDL 的音频回调里，调 `avcodec_decode_audio4` 解出 PCM，解码器输出的是浮点平面格式 FLTP，SDL 只认 S16 交错格式，还要走一次 `swr_convert` 重采样才能送去播放。

编码这边，`savevideofilethread.cpp` 第 375 行设置了 `c->thread_count = 4`，开 4 个 CPU 线程跑 H.264 软编，配了 `preset=superfast` 和 `tune=zerolatency` 用来降低编码延迟。

## 6. 录制视频是多线程，说一下线程同步

录制涉及三个线程：视频采集线程 `PicInPic_Read`、音频采集线程 `Audio_Read`、编码写文件线程 `SaveVideoFileThread`，整体是生产者-消费者模型，前两个是生产者，编码线程是消费者。

线程同步用的是 `QMutex`。`savevideofilethread.h` 里定义了两把锁，`m_videoMutex` 保护视频队列 `m_videoDataList`，`m_audioMutex` 保护音频队列 `m_audioDataList`。视频采集线程每采到一帧，调 `videoDataQuene_Input` 加锁、append、解锁；音频采集线程同理。编码线程在 `run()` 里调 `videoDataQuene_get` 和 `audioDataQuene_get`，同样加锁取走数据再解锁。这样两侧对同一个队列的读写就不会冲突。

编码线程拿到视频帧和音频数据后，还要解决音视频时序对齐的问题。它比较视频帧和音频数据各自的时间戳，哪路时间戳更早就先编哪路，保证写进文件的帧是按时间顺序交错排列的。如果视频队列里暂时没有新帧，编码线程会复用上一帧继续编，保证视频流不断，等待新帧到来再恢复正常。

## 7. 项目用了哪种 IPC

项目的 IPC 用的是 TCP Socket。客户端和服务端是两个独立进程，之间所有通信——登录、注册、上传元信息、文件块传输、下载、历史查询——都封装成 `packdef.h` 里定义的结构体，通过 TCP 发送，没有用 HTTP。

客户端内部的线程间通信用的是 Qt 的 Signal/Slot。代码里的 connect 调用都没有显式指定第 5 个参数，默认是 `Qt::AutoConnection`，Qt 在运行时判断发送方和接收方是否在同一个线程，如果跨线程就自动用 QueuedConnection，把 slot 调用投递到接收方线程的事件队列里执行，不需要手动加锁。

## 8. 音视频的播放怎么做的

**整体架构**：主线程（`VideoPlayer::run()`）负责读包（`av_read_frame`），分发到视频包队列（`videoq`）和音频包队列（`audioq`）；视频解码在独立的 SDL 线程（`video_thread`）里跑；音频解码在 SDL 的音频回调（`audio_callback`）里跑，由 SDL 的混音线程驱动。

**视频解码与渲染**：`video_thread` 从 `videoq` 取包，调用 `avcodec_decode_video2` 解出 YUV 帧，再用 `sws_scale`（SWS_BICUBIC 插值）把 YUV420P 转成 RGB32，最终 emit 信号给 `MyOpenGLWidget`，用 OpenGL 纹理上传到 GPU 渲染，比 QPainter 方案效率更高。

**音频解码与播放**：`avcodec_decode_audio4` 解码出 FLTP 格式 PCM，经 `swr_convert` 重采样为 S16SYS 格式存入 `audio_buf`。SDL 以 1024 samples（约 23ms）为单位回调 `audio_callback`，里面循环调用 `audio_decode_frame` 填充 stream 缓冲区，通过 `SDL_MixAudioFormat` 混音后输出到扬声器。

**音视频同步**：以音频时钟（`audio_clock`）为主时钟，`video_thread` 里在每帧显示前检查 `video_pts <= audio_pts`，若视频超前则 `SDL_Delay(5)` 等音频追上（`videoplayer.cpp` 第 100-108 行）。没有用 DTS 做复杂同步算法，依赖音频输出的稳定性来驱动视频节拍。

## 9. 点击视频观看完后退回上一个 UI，怎么自动刷新和推荐？怎么通知客户端刷新？

**整体流程**：客户端登录成功后，`slot_loginRs` 里直接发一个 `STRU_DOWNLOAD_RQ` 给服务端请求视频列表。服务端的 `DownloadRq` 收到后执行这条 SQL：

```sql
SELECT videoId, picName, picPath, rtmp FROM t_VideoInfo
WHERE videoId NOT IN (SELECT videoId FROM t_UserRecv WHERE userId = ?)
ORDER BY hotdegree DESC LIMIT 0, 10;
```

用 `NOT IN` 子查询把用户已经收到过的视频排除掉，剩下的按热度倒序取 10 条。查完之后把这些 videoId 写入 `t_UserRecv` 表标记为已推送，然后把每条视频对应的 GIF 缩略图文件以 1024 字节为单位分块发给客户端。客户端 `slot_fileblockrq` 把数据写进本地临时文件，等文件接收完整后，把 GIF 加载到对应的 `MovieLabel` 按钮上，同时把 RTMP 播放地址存在按钮里，用户点击后直接用这个地址拉流播放。

**刷新**：代码里没有观看结束后自动触发刷新的逻辑，`playerdialog.cpp` 的定时器检测到播放结束只是重置了 UI 状态，没有发任何网络请求。刷新依赖用户主动点击刷新按钮，`on_pb_fresh_clicked` 里发一个新的 `STRU_DOWNLOAD_RQ`，服务端再走一遍上面的流程返回新的未推送视频。如果用户把所有视频都看完了，服务端会清空 `t_UserRecv` 表，让视频重新变成未推送状态。

**推荐**：推荐逻辑就是热度排序，`hotdegree` 字段越高越靠前。注册时用户填了 8 类兴趣标签，视频上传时也带了对应的兴趣标签，但当前的下载 SQL 里并没有用兴趣标签做过滤，这块是预留的扩展能力。

**通知客户端刷新**：项目没有服务端主动推送，都是客户端主动请求。如果要做推送，服务端维护了 `m_mapIDToUserFD` 这个 `userId → fd` 的映射表，有新内容时可以遍历在线用户直接 `SendData` 推通知包。

## 10. 缩略图怎么实现的

缩略图是 GIF 动图，生成逻辑在 `uploaddialog.cpp` 的 `SaveVideoJpg` 函数里，不管是本地选取的视频还是录制完的视频，上传前都走这同一套流程。

具体做法是用 FFmpeg 打开视频文件，遍历数据包，筛选出前 6 个 I 帧（判断条件是 `pFrame->key_frame == 1 && pFrame->pict_type == AV_PICTURE_TYPE_I`），每帧用 `sws_scale` 转成 RGB32 后包成 `QImage`，缩放到 640×320，保存成 `00.jpg`、`01.jpg`……`05.jpg` 存到本地 `./temp/` 目录。6 帧提取完之后，用 `QProcess` 启动一个 cmd 进程执行 FFmpeg 命令：

```
ffmpeg -r 2 -i temp\%02d.jpg temp\<时间戳>.gif
```

`-r 2` 表示 2 帧每秒，把 6 张 JPEG 合成一个约 3 秒的动态 GIF。

上传时先把 GIF 文件发给服务端，再发视频文件，发视频的 `STRU_UPLOAD_RQ` 包里的 `m_szGifName` 字段带上 GIF 的文件名，服务端把两者都存到磁盘，GIF 路径写入 `t_VideoInfo.picPath`。客户端请求上传历史时，服务端把 `picPath` 下发，客户端用 `MovieLabel` 控件加载 GIF 显示在视频列表里。
