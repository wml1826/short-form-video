## 精简版（150 字以内，适合简历格子）

**MediaPlay —— 基于 C++/Qt/FFmpeg 的全栈短视频与直播平台**

基于 C++ 和 Qt 开发的全栈多媒体平台，支持用户注册登录、本地/在线视频播放、RTMP 直播推拉流、画中画录制、文件上传与内容推荐。客户端基于 FFmpeg 4.2 + SDL2 实现音视频播放管线，基于 OpenCV 采集摄像头、FFmpeg 编码 H.264/AAC 输出；服务端基于 epoll + 动态线程池 + MySQL + Redis 实现高并发业务处理，自定义 TCP 二进制协议完成客户端-服务端通信。

**技术栈：** C++ · Qt 5 · FFmpeg 4.2 · SDL2 · OpenCV · OpenGL · MySQL · Redis · Epoll · 多线程编程 · 自定义 TCP 协议 · RTMP 流媒体

## 完整版（项目亮点 8 条，适合简历详细描述区域）

**项目简介：**
基于 C++ 和 Qt 开发的全栈多媒体平台，覆盖用户系统、本地/在线视频播放、RTMP 直播推拉流、屏幕+摄像头画中画录制、视频分块上传与内容推荐等全链路功能。

**项目亮点：**

1. **自定义 TCP 二进制协议与事件驱动网络架构**
   设计基于 **epoll（LT 模式）** 的事件驱动网络框架，采用 **EPOLLONESHOT** 确保同一 socket 串行处理，避免多线程并发竞态。自定义二进制应用层协议（4 字节包长 + 4 字节协议类型 + 数据体），通过 **函数指针映射表** 将 18 种协议包分发到对应业务处理器，实现网络层与业务层完全解耦。

2. **自研动态线程池（自动扩缩容）**
   基于 **生产者-消费者模型** 和 **环形队列（容量 50000）** 实现动态线程池，初始 10 个工作线程，上限 200。管理线程每 10 秒监控负载：当待处理任务 > 空闲线程数或忙碌比 ≥ 80% 时自动扩容，忙碌线程 < 存活线程 1/3 时自动缩容，兼顾吞吐与资源开销。

3. **FFmpeg + SDL2 音视频播放引擎**
   基于 FFmpeg 4.2 实现完整播放管线：av_read_frame → 音视频流分离 → 多线程解码。SDL2 音频回调驱动播放时序，基于 **PTS 时间戳 + 视频时钟补偿算法**（含 repeat_pict 帧重复修正）实现精确音视频同步（audio master 模式）。支持播放/暂停/停止状态机、seek 跳转，使用 **OpenGL** 硬件加速视频帧渲染。

4. **RTMP 直播系统（推流 + 拉流）**
   实现完整直播功能：主播端通过 FFmpeg（avcodec_send_frame / avcodec_receive_packet 新版编码 API）对摄像头+音频实时采集并编码为 H.264/AAC，FLV 封装后通过 **RTMP 协议** 推送至流媒体服务器；编码管线同时支持本地文件输出和 RTMP 推流（仅替换 filename 参数）。服务端管理直播元数据（streamKey、t_LiveStream 状态表），客户端支持直播间列表查询（JOIN 用户表展示主播名）和一键拉流观看。

5. **画中画录制与编码输出**
   通过 **OpenCV VideoCapture** 采集摄像头画面，Qt 截取桌面帧，叠加为画中画效果。基于 FFmpeg 新版编码 API 完成 H.264 视频 + AAC 音频编码与 FLV 封装，支持屏幕录制和摄像头录制两种模式。

6. **视频分块上传与预览动图**
   大文件按 1KB 分块上传，服务端边收边写。上传前通过 FFmpeg 提取 I 帧生成 GIF 预览动图，提升浏览体验。支持 8 种爱好分类标签用于内容检索。

7. **基于 Redis + MySQL 的轻量推荐系统**
   Redis 缓存视频热度值（hotdegree），服务端按热度降序推荐；通过 t_UserRecv 排重表确保用户不重复观看同一视频，单次推荐上限 10 条，平衡体验与性能。

8. **数据持久化与用户系统**
   MySQL 设计 4 张核心表（t_UserData / t_VideoInfo / t_LiveStream / t_UserRecv），封装为线程安全的数据库操作层。支持完整注册/登录流程，注册时自动创建用户专属存储目录。

## 面试追问与回答要点

项目详情感兴趣的面试官可能会追问，提前准备这些：

| 问题 | 回答要点 |
|------|---------|
| **为什么用 epoll 而不是 select/poll？** | epoll 无监听上限、O(1) 事件通知、支持 EPOLLONESHOT 避免多线程并发、支持 LT/ET 模式 |
| **EPOLLONESHOT 解决了什么问题？** | 同一 socket 被多个线程池 worker 同时处理，导致数据乱序或并发竞态。ONESHOT 确保一次事件只被一个线程处理 |
| **音视频同步怎么做？** | Audio master：音频为主时钟，视频线程循环比对 video_pts 与 audio_clock，视频超前则等待，落后则丢帧追赶 |
| **新版编码 API 和旧版有什么区别？** | 旧版 avcodec_encode_video2 已废弃，新版采用 avcodec_send_frame + avcodec_receive_packet 分离输入输出，支持零拷贝和内部分片 |
| **线程池怎么动态扩缩容？** | 管理线程每 10s 检查：队列待处理 > 空闲线程 || 忙碌比 ≥ 80% → 扩容；忙碌 < 存活/3 → 缩容 |
| **epoll 为什么用 LT 模式？** | LT 更简单、编程模型与阻塞 IO 一致、不易丢事件、适合入门（项目中注释也有体现） |
| **推流和录制用同一套代码？** | 是的，SaveVideoFileThread 接收 STRU_AV_FORMAT，fileName 传本地路径则录文件，传 rtmp:// URL 则推流 |
| **Redis 存了哪些数据？** | 视频热度值（hotdegree）和分类标签（food/funny 等 8 类），用于排序推荐 |
| **推荐算法怎么做的？** | 按热度降序 + 排重表过滤已看 + 一次最多 10 条，简单但够用 |
| **客户端和服务端通信协议设计？** | 4 字节包长前缀（解决粘包）+ 4 字节 PackType（路由到处理器）+ 结构体数据（定长/可计算） |
